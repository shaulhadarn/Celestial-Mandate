Changelog - Stellaris Imperium
================================

2026-02-24: Fixed mobile race intro buttons — added #race-intro-overlay to preventDefault exclusions
- Root cause: Race intro "Continue" and "Begin Your Reign" buttons were unresponsive on mobile because #race-intro-overlay is outside #ui-layer and was blocked by preventDefault()
- Fix: Modified touchmove and touchstart listeners in renderer.js to also exclude #race-intro-overlay from preventDefault()
- Result: All race intro buttons now respond to touch on mobile devices while preserving 3D canvas interactions

2026-02-24: Fixed mobile creation screen buttons — added #creation-screen and #loading-screen to preventDefault exclusions
- Root cause: Species design "Next Step" buttons were unresponsive on mobile because #creation-screen is outside #ui-layer and was blocked by preventDefault()
- Fix: Modified touchmove and touchstart listeners in renderer.js to also exclude #creation-screen and #loading-screen from preventDefault()
- Result: All UI buttons in creation screens now respond to touch on mobile devices while preserving 3D canvas interactions

2026-02-24: Fixed mobile splash screen buttons — added #splash-screen to preventDefault exclusions
- Root cause: After fixing galaxy touch interactions, preventDefault() was blocking touch events on splash screen buttons because #splash-screen is outside #ui-layer
- Fix: Modified touchmove and touchstart listeners in renderer.js to also exclude #splash-screen from preventDefault()
- Result: Splash screen "New Game" button now responds to touch on mobile devices while preserving galaxy drag functionality

2026-02-24: Fixed mobile galaxy touch — drag/pan/zoom completely broken on mobile
- Root cause 1: touchmove/touchstart listeners were registered as passive:true (browser default), so e.preventDefault() was never called, letting the browser consume touch events before OrbitControls could see them
- Root cause 2: OrbitControls touch.ONE was set to ROTATE (orbiting a sphere), which is useless for a top-down galaxy map — should be PAN so one finger drags the map
- Root cause 3: tap distance threshold was 20px, too small for mobile fat fingers causing star taps to never register
- `src/visuals/renderer.js`:
  - touchmove and touchstart listeners now use `{ passive: false }` so e.preventDefault() actually runs
  - preventDefault() is skipped when the touch target is inside #ui-layer (so UI buttons still work)
  - Tap detection threshold increased from 20px → 30px for mobile usability
- `src/Game.js`:
  - OrbitControls touch.ONE changed from ROTATE → PAN (one finger drags the galaxy map)
  - OrbitControls touch.TWO set to DOLLY_PAN (pinch-to-zoom + two-finger pan)
  - screenSpacePanning = true so panning moves in screen XY plane (feels like dragging a map)

2026-02-22: Fix mobile galaxy drag flicker, square flashes, and light-ray artifacts
- Root cause 1: duplicate post-processing render path in R3F loop + renderer loop caused unstable composer output on mobile during drag

- `src/Game.js`:
  - Disabled EffectComposer on mobile and set global composer to null on mobile
  - Removed per-frame composer render from R3F `useFrame` loop to avoid duplicate render pipeline
  - Reduced mobile canvas load (`dpr` capped to `[1, 1.25]`, antialias disabled on mobile)
- `src/visuals/visuals_galaxy.js`:
  - Added mobile detection
  - Reduced outer star halo opacity/size/pulse amplitude on mobile
  - Disabled heavy additive nebula sprite layers on mobile (desktop unchanged)
  - Disabled nebula sprite scale animation on mobile
- `src/visuals/visuals_system.js`:
  - Added mobile detection
  - Reduced system star glow opacity/scale on mobile
  - Reduced planet glow sprite opacity/scale multiplier on mobile
  - Disabled heavy additive background nebula sprites on mobile (desktop unchanged)

2026-02-22: Fix mobile white square / unrendered area glitches in 3D view
- Root cause: EffectComposer uses HalfFloat internal render targets which mobile GPUs don't support cleanly
- Fix: on mobile (userAgent or width<=768), composer is set to null — no EffectComposer created at all
- Render loop already has fallback: when composer===null it calls renderer.render(scene,camera) directly
- renderer.outputColorSpace = SRGBColorSpace is set directly on the renderer for correct colors without OutputPass
- applyGraphicsConfig() and resize handler both guarded with !isMobile && composer checks
- Desktop bloom/OutputPass pipeline unchanged
